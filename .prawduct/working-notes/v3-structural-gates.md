# V3: Structural Gates on a Principles Foundation

Created: 2026-02-22
Branch: `v3-structural-gates`
Parent: `principles-based-architecture` (v2 experiment)
Status: Discovery/Planning

## Motivation

The v2 experiment (principles-based architecture) correctly diagnosed three problems with the v1 process engine:
1. Learning loop was write-only (observations filed, never read)
2. Governance complexity was self-compounding (3,280 lines of Python, 8 hooks)
3. Runtime coupling to framework clone made distribution and parallel work impossible

The v2 solved problems 1 and 3 well. But its first real test — building Hum chunk 1 — exposed a critical gap: **Claude did not invoke the Critic, despite principles explicitly requiring it.** 15 findings went undetected. The session skipped reading `methodology/building.md` entirely, never considered independent review, and self-declared the chunk complete.

This was predicted by the v2 risk assessment ("Principle violations under pressure", "Critic invocation forgetting") but the experiment chose to test whether principles alone could carry the load. The answer is: **no, not for process interruptions that halt build momentum.**

## Core Insight

There is an asymmetry in what principles can govern:

| Category | Example behaviors | Principles sufficient? |
|----------|------------------|----------------------|
| **How to work** | Test quality, scope discipline, spec fidelity, accessibility | Yes — Claude follows these reliably |
| **When to interrupt** | Invoke independent reviewer, read methodology before starting, reflect before ending | No — momentum and helpfulness training override |

The v1 system treated everything as "needs enforcement" (8 hooks, 3,280 lines). The v2 treated everything as "principles suffice" (2 hooks, 60 lines). Neither is right.

The v3 thesis: **principles for judgment, structural gates for interruptions.** Enforce the 2-3 behaviors Claude demonstrably can't self-regulate. Trust judgment for everything else.

## Critical Requirement: Multi-Agent Parallelism

Multiple agents must be able to work on different product repos simultaneously without clobbering each other. This is non-negotiable and shapes the entire architecture.

**What this means:**
- No shared mutable state between agents (no `.session-governance.json`, no `.active-products/` registry, no PID-based session tracking)
- Product repos must be fully self-contained (own CLAUDE.md, own hooks, own Critic instructions)
- No runtime dependency on a framework clone (`framework-path` is gone and stays gone)
- The framework is a **generator** (produces self-contained repos) not a **runtime** (that repos depend on)

**What this rules out:**
- Any hook that resolves paths back to a framework repo at runtime
- Any shared state file that multiple agents might read/write
- Any PID-based session isolation (the v1 approach)

**What this enables:**
- Agent A builds product X while Agent B builds product Y — completely independent
- Multiple agents working on the same framework repo via git worktrees
- Products work anywhere Claude Code runs, no framework clone needed
- Natural distribution: `prawduct init` generates everything, then the product is free-standing

## What Needs to Change (from v2 baseline)

### Keep (v2 got these right)
- 22 principles inline in CLAUDE.md
- Narrative methodology guides (discovery, planning, building, reflection)
- learnings.md as living document
- Session reflection enforcement (stop hook)
- Critic as separate agent (independent review)

### Add (proven necessary by chunk 1 test)
1. **Critic gate** — Structural enforcement that independent review happens before a chunk is declared complete. The specific mechanism is TBD but the requirement is: you cannot proceed past a chunk boundary without Critic evidence.

2. **Phase-transition nudge** — When entering the build phase, actively prompt reading `methodology/building.md`. The chunk 1 session correctly read discovery.md and planning.md; building was the one that got skipped.

3. **Per-prompt governance visibility** (optional) — Lightweight context injection showing current state ("Build phase, chunk 2/6, Critic review: pending"). The v1 prompt hook was purely informational (never blocked) but was valuable for maintaining awareness through compaction.

### Design (new requirement)
4. **Product repo self-containment** — Products generated by prawduct must carry their own:
   - CLAUDE.md with principles, routing, methodology content
   - Hooks (reflection + Critic gate)
   - Critic instructions (either embedded or as a file in .prawduct/)
   - Everything needed to build without the framework repo

### Preserve (from v2, unchanged)
- Old governance module and agent skills remain on disk for reference
- Methodology guides continue to be essays, not checklists
- Minimal hook count (target: 3-4, not 8)
- Total enforcement code: target ~150-200 lines (not 3,280)

## Design Decisions

### D1: Critic instructions — ship a condensed product-review file

The full `agents/critic/SKILL.md` is 245 lines covering 10 checks, framework review cycles, and extension guidance. Product builds only need 6 checks (Spec Compliance, Test Integrity, Scope Discipline, Proportionality, Coherence, Learning/Observability). These condense to ~300-400 words.

**Decision**: Generate `.prawduct/critic-review.md` (~400 words) in product repos containing just the product-relevant checks, output format, and the files to read. The Critic agent prompt says "Read .prawduct/critic-review.md for your review instructions." This gives genuine independence (separate agent, separate context) without requiring framework access.

**Trade-off**: Product Critic instructions may drift from framework updates. Accepted: the cost of staleness is lower than the cost of runtime framework dependency. Products can regenerate critic-review.md from a newer framework version if needed.

### D2: Critic gate — stop hook checks for findings file

The gate must be simple, local, and hard to game.

**Decision**: The stop hook (already enforcing reflection) additionally checks: if `.prawduct/artifacts/build-plan.md` exists AND source files were modified, then `.prawduct/.critic-findings.json` must exist and be recent (within this session). If not, block with a message directing the agent to invoke the Critic.

**Why stop hook, not PreToolUse**: PreToolUse gates require knowing what "next chunk edits" look like vs. "current chunk fixes" — that's complex classification the v1 system struggled with. The stop hook is simpler: you can't end a session where you built code without having run the Critic. This catches 100% of the chunk-1-findings failure mode.

**What "recent" means**: The findings file's mtime is after the session-start marker. This prevents stale findings from a previous session counting.

### D3: Phase-transition — CLAUDE.md instruction, not a hook

The chunk 1 failure was: Claude didn't read building.md when entering the build phase. A hook could detect this, but it requires knowing "this is the first build-phase edit" — which means tracking state about what phase you're in.

**Decision**: Strengthen the CLAUDE.md routing instruction to make it unmissable. Add a bold, explicit instruction in the Build Cycle section: "BEFORE writing any code: read methodology/building.md." Also strengthen the building.md Critic section to make it clear this is per-chunk, not optional.

**Why not a hook**: Adding phase-detection logic is the first step toward rebuilding the orchestrator. The v1 had this (stage tracking in session-governance.json) and it was a major complexity source. If the CLAUDE.md instruction proves insufficient, we can add a hook later — but test the lightweight approach first.

### D4: Methodology in product repos — condensed inline + full files

Product repos need methodology guidance but shouldn't require the framework repo.

**Decision**: Product CLAUDE.md includes condensed methodology (key points from each guide, ~500 words total). Full methodology files are optionally copied to `.prawduct/methodology/` during init for reference. The condensed version is sufficient for operation; the full version adds depth.

**Why both**: The condensed version stays in CLAUDE.md (always loaded, survives compaction). The full files provide depth for complex situations. Either works alone.

### D5: Learnings portability — manual, not automated

**Decision**: Product learnings stay in the product repo. Framework learnings stay in the framework repo. Cross-pollination is manual (user reviews product learnings and updates framework if a pattern is general). No automated extraction.

**Why**: Automated extraction creates a shared-state channel between repos, violating multi-agent safety. Manual review also applies human judgment about whether a product-specific learning generalizes.

## Implementation Plan

### Architecture

```
Framework repo (prawduct/)           Product repos (generated)
├── CLAUDE.md (framework dev)        ├── CLAUDE.md (self-contained)
├── methodology/                     ├── .prawduct/
│   ├── discovery.md                 │   ├── project-state.yaml
│   ├── planning.md                  │   ├── learnings.md
│   ├── building.md                  │   ├── critic-review.md
│   └── reflection.md                │   ├── methodology/  (optional full copies)
├── agents/critic/SKILL.md           │   ├── artifacts/
├── tools/reflection-hook            │   └── .session-reflected
├── tools/prawduct-hook              ├── .claude/settings.json (local hooks)
└── .prawduct/ (framework state)     └── src/ ...
```

Multi-agent safety: each product repo is fully independent. No paths point outside the repo. No shared state files. Agent A in product X and Agent B in product Y never touch the same file.

### Chunks

**Chunk 1: Enhanced reflection-hook with Critic gate** (framework)
- Modify `tools/reflection-hook` to add Critic-gate logic to the stop subcommand
- When build-plan exists AND source files changed AND no recent critic-findings → block
- Add `record-review` subcommand that the Critic agent calls to mark review complete
- Update `.claude/settings.json` if new hook subcommands need new matchers
- ~80-100 lines total (up from 60)
- Test: verify hook blocks without findings, passes with findings

**Chunk 2: Condensed critic-review.md for products**
- Write the product-focused Critic instructions (~400 words)
- 6 checks (Spec Compliance, Test Integrity, Scope Discipline, Proportionality, Coherence, Learning)
- Output format, severity levels, files to read
- Invocation prompt template (what the builder says to spawn the Critic agent)
- Place in `templates/critic-review.md` (source) and test it by running a Critic review using only this file

**Chunk 3: Product CLAUDE.md template**
- Write the self-contained CLAUDE.md for product repos
- 22 principles inline (same as framework)
- Condensed methodology (~500 words covering discovery → planning → building → reflection essentials)
- Routing section (new product, returning user, first contact)
- Explicit Critic invocation instructions with per-chunk requirement
- Learning loop instructions
- Compact instructions
- Place in `templates/product-claude.md`

**Chunk 4: Product hook and settings generation**
- Create `tools/product-hook` — the hook script that ships with product repos
- Handles: clear (reset session state), stop (reflection + critic gate)
- No framework-path references
- Create settings.json template for product repos
- Test: hook works in a standalone directory with no framework present

**Chunk 5: Updated prawduct-init**
- Update `tools/prawduct-init.py` to generate self-contained product repos
- Generate: CLAUDE.md (from template), settings.json, product-hook, critic-review.md, .prawduct/ structure
- Remove framework-path generation
- Remove framework-version tracking (no longer relevant)
- Test: generated repo works independently

**Chunk 6: Framework CLAUDE.md refinement**
- Strengthen build-phase transition instructions
- Add explicit "Read building.md BEFORE writing any code" with emphasis
- Strengthen Critic invocation language at chunk boundaries
- Update compact instructions
- Update project-structure.md for new file layout

### Governance Checkpoints

- After Chunk 2: Review — are the condensed Critic instructions sufficient? Test with a real review.
- After Chunk 5: Full integration test — generate a product repo, verify it works standalone.

## Open Questions (remaining)

1. **Hook format**: Should `tools/product-hook` be bash (portable, simple) or Python (more capability)? Leaning bash for simplicity and because the v2 reflection-hook proved bash is sufficient for this.

2. **Methodology file copying**: Should `prawduct-init` copy the full methodology files into `.prawduct/methodology/`? Or is the condensed CLAUDE.md section sufficient? Could offer as a flag: `--full-methodology`.

3. **Critic findings format**: Should products use the same `.critic-findings.json` format as the framework, or something simpler? Simpler is easier to gate-check; same format enables tooling reuse.

## Risks

- **Gate circumvention**: Claude might write perfunctory critic-findings.json without actually running the Critic. Mitigation: the stop hook checks for the file, but the CLAUDE.md instructions and building methodology both emphasize genuine independent review. The gate prevents forgetting; it can't prevent bad faith. But bad faith is a much rarer failure mode than forgetting.
- **Product CLAUDE.md staleness**: Products carry their own principles and may diverge from framework updates. Mitigation: version tracking in generated comments, but accept divergence as the cost of independence.
- **Over-correction slide**: Adding structural gates could restart the complexity cycle. Mitigation: strict budget (max 4 hooks, max 200 lines enforcement code). This risk is in learnings.md. Any proposal to add a 5th hook must clear a high bar.
- **Condensed methodology insufficiency**: The ~500-word condensed methodology might miss important guidance. Mitigation: offer full methodology files as optional copies; strengthen based on testing.
